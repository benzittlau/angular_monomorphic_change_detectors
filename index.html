One of the major roles modern javascript frameworks serve is to allow you to combine data with a template to create a DOM structure. There was a time when it might have been useful simply to do this once, but modern javascript apps often have changing data and state, and it is necessary to make sure the interface reflects any changes to that data. Change detection is the first part of that process; allowing the framework to be aware that changes have occurred, so they can handle those changes appropriately. We won't be looking at how those actual updates are applied in this talk, if you're interested in that please see my talk from last year on Virtual DOM.

So why should we care about change detection?

DOM changes are slow, so it's hard to make an app fast

Different approaches used by the different frameworks

No "right" answer

I personally find the different ways in which frameworks handle the challenges of change detection of DOM mutation incredibly interesting. Making a web application "feel" fast is very hard (as anyone who's attempted to make an interactive application without the modern tools can likely attest). Each framework has taken a different approach, and neither of these approaches has "won". As the frameworks are converging towards similar concepts in terms of components and data and action flows, their change detection and update approaches have remained very different. How the framework handles these questions has direct impact on how the user of the framework has to work, so understanding these differences gives a better appreciation for the usage differences between the frameworks.

To summarize...

Change detection is how frameworks know they need to update the view

Each framework uses a different strategy

Angular's strategy is to check each components state each "tick" to see if it's changed

But how do you make it fast?

Source
While this may seem pretty fast when you're only looking at a single component, applications will often be made up of many, many components in a tree structure like that shown here. Each of those components must check it's own data to see if it's changed (this isn't strictly true in Angular, but is true by default). Because this needs to happen every cycle, we need it to be incredibly fast.
